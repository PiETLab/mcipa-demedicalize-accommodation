# Participant Instruction Sheet: July 22 ActionForge Feedback Session

## Purpose of the session

On July 22, Sabine, a collaborator on this project, will lead a virtual gathering session for participants connected with the MCIPA project. The session will help us assess the content of the *Demedicalize Accommodation* advocacy paper and  test the ActionForge approach.

ActionForge is a project concept that uses GitHub as a public place to organize, preserve, and improve project materials. You do **not** need to know how to code or use GitHub as a software tool. For this session, GitHub will simply be used as a structured way to submit feedback for the project team to review.

Please see the short project concept note: [[Policy and Admin/The ActionForge Project Concept]].

## What we are asking you to do

**Before the session**:

- Read the [[Advocacy Paper|Advocacy Paper]].
- Notice anything you think may need discussion, clarification, revision, or further attention.
- Choose one point you might want to raise as feedback. 

**Your feedback might be about**:

- something in the paper that is unclear;
- a point you agree with but think needs stronger wording;
- a concern about language, framing, accessibility, or inclusion;
- something important that seems missing;
- a suggestion for revision;
- a question the paper should address;
- a substantive disagreement or complication.

**During the session**:

- Sabine will explain how the ActionForge feedback process works.
- You will be guided through a simple form on the project website for submitting feedback through GitHub Issues (URL will be provided at the start of the session).
- You will write your feedback as an “issue,” which means a structured comment, question, concern, or suggested change for the project team to consider.
- You do not need to use technical language.
- You do not need to solve the issue yourself. It is enough to clearly identify what you noticed and why it matters.

We are also testing whether this feedback process is accessible and comfortable to use. If GitHub does not work well for you, including with a screen reader or keyboard navigation, Sabine will help you submit your feedback another way.

## Why we are using GitHub Issues

For this session, GitHub Issues is being used as a feedback and governance tool, not as a coding tool.

It helps us test whether participants can submit feedback in a way that is:

- visible to the project team;
- organized in one place;
- easier to track than scattered email comments;
- connected to future revisions of the paper;
- part of a transparent process for deciding what changes may be needed.

## What you may get out of the session

By participating, you will:

- help shape the next version of the call-to-action paper;
- learn one simple way to submit structured feedback to a public project;
- help us test whether ActionForge is usable and appropriate for this kind of community-based work;
- contribute to a governance process where feedback can be documented, reviewed, and acted on.

## Important notes

- No coding experience is expected.
- You will be guided through the process step by step.
- The focus is on your ideas, concerns, and feedback, not on technical skill.
- The session is part of the approved MCIPA research process for paid participants.
- Please only share what you are comfortable sharing in this setting.
